# 修复日志 - 2026-05-12

## 背景

用户报告: **"前端显示有问题,没有正确的显示出测评结果"**

经测试 (`curl /api/student/submission/result/6` + 直查 PG),定位到这是 3 个独立 bug 叠加的结果,不是单点问题。

## 现象 vs Root Cause

| Bug | 现象 | Root Cause |
|-----|------|-----------|
| **A** | `/api/student/submission/result/6` 返回 `id=32, llmScore=0, llmReview=null` (空记录),而 DB 里 `evaluation_result` 表对同一 `submission_id=6` 实际有 4 条,最新一条 `id=36, llmScore=8` 含完整 markdown 评审 | `ResultService` 用 `ConcurrentHashMap` 做查询缓存,第一次查询时(LLM 还没跑完)把空占位行缓存进去,后续重评写库的真实结果 evictCache 从未被调用 → 永远命中旧缓存 |
| **B** | 所有 `submission.submitTime` 在 API 响应里都是 `null`,前端显示 "-" | `SubmissionMapper.update` 的 SQL 是 `UPDATE submission SET submit_time=#{submitTime}, status=#{status} WHERE id=#{id}`,但 `TaskProcessorService.updateSubmissionStatus` 只 set 了 id+status,`submitTime` 字段为 null → 每次状态变更(待评测→评测中→完成)都把 `submit_time` 抹成 NULL |
| **C** | LLM 评审在前端以 `<pre>` 显示 markdown 源码,`**加粗**` / `## 标题` / `- 列表项` 等符号原样可见,不解析 | `SubmissionDetailView.vue` 没有 markdown 渲染器,直接把字符串塞进 `<pre class="llm-review">{{ ... }}</pre>` |

## 修复

### Bug A: ResultService 缓存不刷新

#### 修改文件
- `backend/src/main/java/com/javaevaluation/service/TaskProcessorService.java`

#### 变更点
1. 注入 `ResultService`:
   ```java
   private final ResultService resultService;
   ```
2. `saveEvaluationResult` 写库后立即驱逐缓存:
   ```java
   evaluationResultMapper.insert(evaluationResult);
   resultService.evictCache(task.getSubmissionId());
   ```
3. 删掉因下面 Bug B 修复而不再使用的 `Submission` 实体 import

`ResultService.evictCache(Integer)` 早已存在,只是没人调用。无需改 `ResultService` 本身。

### Bug B: `updateSubmissionStatus` 把 submit_time 抹成 NULL

#### 修改文件
- `backend/src/main/java/com/javaevaluation/mapper/SubmissionMapper.java`
- `backend/src/main/java/com/javaevaluation/service/TaskProcessorService.java`

#### 变更点
1. `SubmissionMapper` 新增专用方法,只改 `status` 不动 `submit_time`:
   ```java
   @Update("UPDATE submission SET status = #{status} WHERE id = #{id}")
   int updateStatus(@Param("id") Integer id, @Param("status") Integer status);
   ```
2. `TaskProcessorService.updateSubmissionStatus` 改调新方法:
   ```java
   public void updateSubmissionStatus(Integer submissionId, Integer status) {
       submissionMapper.updateStatus(submissionId, status);
   }
   ```
   旧实现 `new Submission().setId().setStatus().update()` 删除。

`SubmissionMapper.update(Submission)` 原方法保留(`CodeSubmitService` 提交时仍在用,且那里 `submitTime` 是有值的)。

### Bug C: LLM 评审 markdown 渲染

#### 修改文件
- `frontend/package.json` + `package-lock.json` (新增依赖)
- `frontend/src/views/student/SubmissionDetailView.vue`

#### 变更点
1. 新增依赖:
   - `markdown-it` (运行时)
   - `@types/markdown-it` (devDependency)
   - 安装命令: `npm i markdown-it && npm i -D @types/markdown-it`
2. 组件改造:
   - import `MarkdownIt from 'markdown-it'`
   - `const md = new MarkdownIt({ html: false, linkify: true, breaks: false })`
   - `const renderedReview = computed(() => evaluation.value?.llmReview ? md.render(evaluation.value.llmReview) : '')`
   - 模板:`<pre class="llm-review">{{ evaluation.llmReview }}</pre>` 改为 `<div class="llm-review markdown-body" v-html="renderedReview"></div>`
3. 样式:增加 `:deep()` 规则覆盖 markdown 元素(h1-h4 字号/边距、ul/ol/li、strong、inline code、code block 暗色背景、blockquote 蓝色左边框),保持与 Element Plus 整体配色一致

`html: false` 防 XSS(LLM 输出不允许 raw HTML);如果未来想支持代码高亮可以考虑接 `highlight.js`。

## 数据修复 (已直接在 PG 执行)

仅回填 `submission.submit_time`,不删除任何 `evaluation_result` 行(用户选择保守方案)。

SQL (`/tmp/backfill_submit_time.sql`):
```sql
UPDATE submission s
SET submit_time = COALESCE(
  (SELECT MAX(created_at) FROM evaluation_result e WHERE e.submission_id = s.id),
  NOW()
)
WHERE s.submit_time IS NULL;
```

影响行数: **6 行** (submission id 1~6 全部回填)。

回填后 DB 状态:
```
 id | student_id | homework_id |        submit_time         | status
----+------------+-------------+----------------------------+--------
  1 |          1 |           5 | 2026-04-28 01:10:37.81781  |      2
  2 |          1 |           8 | 2026-04-25 08:49:55.586363 |      2
  3 |          1 |           9 | 2026-04-25 09:10:40.233925 |      2
  4 |          1 |          10 | 2026-04-25 09:49:06.438514 |      2
  5 |          1 |          11 | 2026-05-11 18:44:32.025765 |      2
  6 |          1 |          12 | 2026-05-11 18:55:20.476205 |      2
```

`evaluation_result` 表里 9 条旧占位行(`llm_review IS NULL`)未清理。Mapper `findBySubmissionId` 用 `ORDER BY created_at DESC LIMIT 1` 选最新一条,正好选中带 LLM 评审的那条,所以保留旧占位行不影响显示。

## 仍未处理 / 未来工作

- 重复 evaluation_result 行(每次重评 insert 新行而非 update 旧行) — Mapper 没有 update 方法。LIMIT 1 + DESC 兜住了,但 DB 行数会随重评次数线性涨。如果要彻底解决:`evaluation_result.submission_id` 加 UNIQUE 约束 + 改用 UPSERT。

- `evaluation_result` 表的 `execution_time` 字段始终是 0 — `TaskProcessorService` 写 `setExecutionTime(0L)` 占位。需要在 `DockerSandboxService` 里实际计时。
