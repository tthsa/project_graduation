# Skill: 反过度设计检查清单

**适用场景**: 任何项目（Web 后端、前端、脚本、微服务）在代码审查或重构时使用
**核心原则**: YAGNI — You Aren't Gonna Need It

---

## 一、常见过度设计症状（技术无关）

### 症状 1: 为 "万一" 引入抽象
```
❌ "以后可能会扩展，先预留接口"
❌ "虽然现在只有1个实现，但先定义接口"
❌ "先做成配置化，以后改起来方便"
```

**判断**: 如果 "以后" 是假设性的，且概率 < 50%，不要做。

### 症状 2: 一行代码的封装
```
❌ 为 framework.send() 封装一个 Service.send()
❌ 为 utils.trim() 封装一个 Helper.trim()
❌ 为 cache.get() 封装一个 CacheService.get()
```

**判断**: 封装后调用链比直接调用还长，就是过度封装。

### 症状 3: 框架特性滥用
```
❌ 2 个字段的配置用 @ConfigurationProperties / 复杂配置对象
❌ 简单条件判断用上策略模式 + 工厂 + 注册表
❌ 单表 CRUD 引入 CQRS / Event Sourcing
❌ 单机脚本引入 Docker + K8s
```

**判断**: 框架特性是为了解决特定复杂度，不是用得越多越好。

### 症状 4: 手写已存在的基础设施
```
❌ 手写 HashMap 做缓存（无过期、无淘汰）
❌ 手写连接池
❌ 手写 JSON 解析
❌ 手写 ORM
```

**判断**: 标准库 / 成熟框架已经有的，不要自己写。

### 症状 5: 重复的定义
```
❌ 同一套 token 解析逻辑在 N 个 Controller 中重复
❌ 同一套权限校验在 N 个 Service 中重复
❌ 同一套响应包装在 N 个文件中重复
```

**判断**: 如果改了这里忘了那里会出 bug，就该提取公共逻辑。

## 二、快速决策框架

面对任何设计选择，连续问自己三个问题：

### Q1: 现在不这么做，最坏会怎样？

| 如果我现在不... | 最坏结果 | 是否可接受？ |
|----------------|---------|------------|
| 不定义接口 | 直接改实现类 | ✅ 1个实现时完全OK |
| 不用配置对象 | 改代码里的 @Value | ✅ 4个字段时完全OK |
| 不手写缓存 | 每次查数据库 | ✅ 流量低时完全OK |
| 不做策略模式 | if-else 多几行 | ✅ 3个分支时完全OK |

→ **如果最坏结果可接受，就不要做。**

### Q2: 改了需求，改哪里更快？

```
场景: 从 RabbitMQ 换成 Kafka

方案A: TaskQueueService.sendTask() 封装
  改1处: 改 Service 里的实现
  但要确认: 所有调用方只通过这个 Service 发消息

方案B: 直接调 rabbitTemplate.convertAndSend()
  改N处: 全局搜索替换
  但 IDE 全局替换只要 10 秒

→ 在这个场景下，封装并没有更快
```

**规则**: 只有当 "封装后的修改点比直接调用更少" 时，封装才有价值。

### Q3: 看这段代码的人，多久能理解？

```
❌ 需要理解 3 个类、2 个接口、1 个工厂，才能看懂一行业务逻辑
✅ 直接从入口到出口，最多跳 2 层
```

**规则**: 理解成本 > 维护成本。如果新人看代码需要画架构图，就过度了。

## 三、各技术栈常见陷阱速查

### Java / Spring Boot

| 特性 | 适用场景 | 不适用场景 |
|------|---------|-----------|
| @ConfigurationProperties | 10+ 配置项 / 多处复用 / 需校验 | <5 个字段，1 处使用 |
| @Cacheable | 任何需要缓存的地方 | 不要用 HashMap 手写 |
| 接口 + Impl | 多实现 / 团队分工 / 需要 Mock | 只有 1 个实现 |
| DTO / VO / BO 分层 | 大型项目 / 团队规范 | 毕业设计 / 个人项目 |
| 事件驱动 / MQ | 异步解耦 / 削峰填谷 | 同步调用即可的场景 |

### 前端 (Vue / React)

| 特性 | 适用场景 | 不适用场景 |
|------|---------|-----------|
| Pinia / Vuex | 跨组件共享状态多 | 只有 2-3 个组件用 props 即可 |
| 组件库封装 | 组件复用 > 3 次 | 只用一次的业务组件 |
| 自定义 Hook | 逻辑复用 > 2 处 | 只在 1 个组件里的逻辑 |
| TypeScript 严格模式 | 团队协作 / 大型项目 | 个人快速原型 |

### 数据库 / 运维

| 特性 | 适用场景 | 不适用场景 |
|------|---------|-----------|
| Redis 集群 | 高并发 / 数据量大 | 日活 < 1000 |
| 分库分表 | 单表 > 1000万 | 数据量 < 100万 |
| Docker | 环境一致性要求高 | 本地开发单机运行 |
| K8s | 多服务 / 自动扩缩容 | 单服务 / 固定流量 |

## 四、修复策略

### 发现过度设计后怎么做

**Step 1: 识别**
```
发现代码里有 "为了抽象而抽象" 的痕迹
↓
问自己: "如果现在删掉这层，会影响什么？"
↓
如果答案是 "几乎不影响"，确认是过度设计
```

**Step 2: 评估影响范围**
```
内部实现改动（如 Service 内联）→ 低风险，可直接改
接口改动（如 URL / 参数变化）→ 中风险，需检查调用方
数据格式改动（如 JSON 结构变化）→ 高风险，需同步改前端
```

**Step 3: 渐进式简化**
```
不要一次性大改，优先改:
1. 纯内部实现（零外部影响）
2. 重复代码提取（减少维护面）
3. 危险代码（如内存泄漏）
4. 接口简化（最后做，需协调前后端）
```

---

## 五、一句话总结

> **"能直接写的，不要封装；能少一个类的，不要多一个；能用框架的，不要手写。"**

扩展性不是提前写出来的，是需求来了之后改出来的。
